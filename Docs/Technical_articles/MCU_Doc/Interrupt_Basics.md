# 割り込みとは何か

ボタンが押されたときなど、いつ起きるか分からない出来事にすぐ反応したい場面があります。このページでは、そのための仕組みである割り込みを、メインの処理との関係や注意点とあわせて説明します。

## イベントを知らせる通知

割り込みは、「何か起きたよ」と周辺回路がCPUに知らせる仕組みです。たとえば、ボタンが押された、UARTで文字を受け取った、タイマーが指定の時間になった、といったときに使われます。

通知を受けると、CPUはいったん今の作業を止めて、登録しておいた処理を行います。終わったら、もとの作業に戻ります。

## ポーリングとの違い

ポーリングでは、メインループが周辺回路の状態を繰り返し確認します。単純で動作を追いやすい方法です。一方で、確認間隔が長いとイベントを見逃す可能性があります。

割り込みでは、イベント側から処理開始を知らせます。すばやく反応しやすい一方、どのタイミングでも呼び出され得るため、共有データや処理時間への注意が必要です。

## 割り込み処理は短くする

割り込み処理が長いと、メイン処理や同じ優先度以下の割り込みを待たせます。割り込み内では、フラグを立てる、受信データをバッファへ移す、カウンターを増やす、といった短い処理に留めます。

特にログ出力は、文字列の整形やUART送信待ちによって予想以上に時間がかかります。割り込み中にログを出すと、その間に次の受信やタイマーイベントが発生してもすぐに処理できず、通信データの取りこぼしや周期のずれにつながります。また、ログを増やしたときだけ動作タイミングが変わり、不具合の再現や原因の特定を難しくすることがあります。

割り込み内では必要な値やエラーをフラグ・カウンターへ記録し、ログ出力や解析はメインループ側で行います。

次の処理は原則として避けます。

- `HAL_Delay()`などの待機
- 終了時刻が読めないループ
- 大量のログ出力
- 重い浮動小数点計算
- ほかの処理と競合する複雑なデータ更新

## 優先度

複数の割り込みが重なったときは、NVICの優先度で実行する順番が決まります。最初は、すべてを高い優先度にする必要はありません。本当に急ぐ処理だけを優先する、と考えると整理しやすくなります。

## 共有する変数

割り込みとメインループの両方から変数を参照する場合、コンパイラによる不適切な最適化を防ぐため`volatile`が必要になることがあります。ただし、複数手順からなる更新を`volatile`だけで安全にはできません。必要に応じて割り込み禁止区間や、用途に合った同期方法を使います。

## まとめ

割り込みは、外部または周辺回路のイベントへすばやく反応する仕組みです。割り込み内の仕事を短くし、重い処理をメインループへ渡す設計が基本になります。
